## 4. 当前判断

截至目前：

- 之前把 batch 语义理解错了，导致对 `exit_code=9` 和 `0 分` 的解释都偏了
- `exit_code=9` 不能单独视为失败
- `v3-v5` 的真正问题是“我方主跑评测失败”
- `v2` 目前仍是已验证的最强版本，真实分数 `607`
- `v6` 说明无 LLM 纯规则也能稳定出分，但当前只有 `371`

## 5. 当前最重要的后续动作

3. 重点优化：
   - 最终答案准确率

## 7. 新发现的后端问题

通过对 `7422408` 的回放与 `err` 日志解码，已经确认：

- 即使是 `admin/跑通测试` 这种高分成功对局
- 在第二个剧本推进到 `stage 8` 后继续 `chat`
- 也会触发后端异常：
  - `Stage.py`
  - `update_npc_question_marks`
  - `KeyError: '8'`

这说明：

- 高阶段继续追问本身就是风险点
- 一部分“我方主跑失败”未必是 AI 逻辑差，也可能是踩中了后端 bug

## 8. 当天收束结论

继续核查后，当前已能明确确认：

- `v7`（线上 `version 8`）我方主跑 `7422058` 已完成，分数是 `407`
- 因此它没有超过 `v2`
- `g2v2ref0311` 的重传 `v2` 已经出现：
  - `7422137 = 407`
  - `7422139 = 607`

这说明当前最大问题不是“缺少更多新版本”，而是：

- 同一份 `v2` 代码本身就有明显分数波动
- 波动来源更像是：
  - 中途问题规划随机
  - 最终答案归纳随机

进一步的回放对比也说明：

- `v7` 额外增加的证据追问，让第一案嫌疑人从 `范敏敏` 偏到了 `叶文潇`
- 这类“更深、更细”的追问目前没有带来稳定增益
- 高分对手 `admin` 的强点是：
  - 问得更多
  - 但问题明显更短、更直接
  - 不是把多个线索塞进一条长问题

因此后续主线调整为：

1. 不再盲目新开版本
2. 先把 `v2` 的问题生成和最终答案生成尽量做得更稳定、更可复现
3. 新版本上线前，优先做重复 batch 验证，确认不是单次幸运高分
