# Game2 Batch 与结果分析机制

更新时间：`2026-04-19`

## 1. 最重要的口径修正

`Game2` 的 batch 不是传统对战。

每个对手 code 对应的一对 `matches`，真正含义是：

- 一场是“对手 code 自己的一次独立跑分”
- 另一场是“我方 code 自己的一次独立跑分”
- 每场里第二个位置的 `score=0` 只是占位

所以 `Game2` 的 batch 应该这样读：

- `my_score` = “我方 code 在第一位置”的那场 raw score
- `opp_score` = “对手 code 在第一位置”的那场 raw score

但如果目标只是判断我方版本强弱，那么还可以进一步简化为：

- `Game2` 本质只看我方自己的 raw score
- “对手 code” 只是为了方便批量制造多份自跑样本

因此：

- 不应该再用 `win/loss/tie` 当主口径
- 不应该把 `Game2` 理解成 Game1 那种双边强弱关系

## 2. 当前已打通的链路

实测确认可用：

- `POST /api/batches/`
- `GET /api/batches/<batch_id>/`
- `GET /api/batches/?game=53&limit=5`
- `GET /api/matches/<match_id>/`
- `GET /api/matches/<match_id>/download/`

这意味着：

- 自动化脚本中断后，仍然可以通过 `batch_id` 补抓结果
- 不需要依赖“当时那次轮询必须成功收口”

## 3. 当前脚本与用途

### 3.1 `Game2/tools/run_batch_eval.py`

用途：

- 读取当前活跃 code
- 从榜单拿若干 code 作为样本载体
- 创建 batch
- 轮询 batch
- 把所有已结束 match 的详情和下载文件落盘
- 自动生成结构化 `analysis.json` / `analysis.md`

关键补充：

- `--batch-id <id>` 可直接回填历史 batch
- 这对“自动化挂了，但 batch 其实早已跑完”的场景非常重要

### 3.2 `Game2/tools/analyze_match.py`

用途：

- 拉取单场 `match`
- 下载完整轨迹
- 解析每个 case 的：
  - 步数
  - 阶段推进
  - 访问 NPC 次数
  - 出示证据次数
  - 最终答案与正确性
  - 性能采样

### 3.3 `Game2/tools/compare_match_runs.py`

用途：

- 对比不同版本或不同样本的 match 目录
- 快速定位：
  - 哪一案在掉 stage
  - 哪些问题是新增的无效噪声
  - 哪些版本只是“问得更多”，并没有问得更准

## 4. `exit_code=9` 的解释

对 `Game2`：

- `exit_code=9` 不能直接等价成失败
- 已经多次观察到 `评测成功 + exit_code=9 + 有完整回放 + 有有效分数`

有效性应该以这些条件判断：

- `match.state`
- `end_state`
- 是否拿到有效 raw score
- 是否能下载轨迹

## 7. 当前最有用的 batch 用法

从现在开始，`Game2` 的 batch 更适合被当成：

- 重复自跑采样器

而不是：

- 胜负评测器

更实用的流程应是：

1. 提交新版本。
2. 用 batch 一次拿多个自跑样本。
3. 只比较我方各个 `my_match_id` 的 raw score、stage、calls。
4. 如果一个版本只在单场上偶然高分，不应直接切主线。
