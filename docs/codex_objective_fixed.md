# Codex 固定优化目标（长期）

该目标用于自动化 codex 迭代，默认不随单轮任务改变。

## 总目标

在不修改 `Ant-Game/` 上游代码的前提下，持续提升 `/www/ai_cpp/` 各版本在蚁洋陷役2中的实战强度与稳定性。

## 硬约束

1. 只能在 `/www` 根仓库内改动，禁止修改 `/www/Ant-Game/`。
2. 优先“算法本质创新 + 结构简化”，避免只做参数微调或无上限堆逻辑。
3. 每轮必须有可运行验证（至少一轮 `autolab_eval.py` 或等价本地评测）。
4. 每轮必须更新文档，记录“改动-结果-风险-下一步”到 `/www/docs/round2_autolab_and_iterations.md`。
5. 不写入或泄露任何 token、密钥、隐私数据。
6. 实验评测必须与生产评测隔离：默认使用 `autolab/runtime/scopes/iter`，且不得触发生产 champion 晋升。
7. 相对强度评估门槛：
   - 两个 AI 的相对强度比较：样本至少 `100` 局，否则仅可视为 smoke。
   - 多个 AI 的排序比较：需要显著更多样本（不低于两两比较门槛），不得用小样本直接下结论。
8. 迭代阶段允许先快速产出多个版本，但“优劣结论”必须谨慎，优先等待自动评测/批量测试稳定后再确认。
9. 若要声明“新 AI 明确优于 k 个老版本”，至少满足以下之一：
   - 与每个目标老版本分别对战 `>=200` 局，且胜率均 `>55%`；
   - 在大样本（如 `>=1000` 局）Elo 评测中，新版本稳定位于榜首。
10. Elo 治理规则：
   - 生产 Elo（`/www/autolab/runtime/latest.json`）是唯一权威排名，可用于 champion 决策；
   - 迭代 Elo（`/www/autolab/runtime/scopes/iter/latest.json`）仅用于候选筛选，不可直接作为最终结论；
   - 不同 scope / 不同对阵池下的 Elo 绝对值不可直接横向比较。
11. 思路同步是强制项：
   - 每轮除更新 `/www/docs/round2_autolab_and_iterations.md` 外，还必须同步更新 `/www/docs/version_strategy_deep_dive_v1_v53.md`；
   - 对应版本条目至少包含：设计假设、关键机制、评测证据、结论标签（`promising/neutral/regression/placeholder`）。
12. 搜索类算法时间上限（硬要求）：
   - 若版本包含搜索/前瞻/rollout/beam/minimax 等，单步决策 CPU 时间必须 `<=200ms`；
   - 必须在代码中实现强制超时截止与保底回退（例如回退到当前最优候选或贪心动作）；
   - 计时必须基于 CPU 时间口径（如 `CLOCK_THREAD_CPUTIME_ID` / 等价实现），不能仅用 wall-clock 代替；
   - 禁止提交“仅靠期望运行速度”而无硬截止的搜索实现。
13. 生产 Elo 判读约束：
   - `gauntlet` 口径下 Elo 强依赖当前 `champion` 与对手集，不得把不同 tag 的 Elo 绝对值直接当作可比强度；
   - 单轮 gauntlet 只能作为候选信号，不能单独构成“版本明显更强”的结论。
14. 生产晋升谨慎原则：
   - 若生产口径出现与近期结论冲突的排序跳变，先做 head-to-head 复验（`>=100` 局）再下结论；
   - 在关键版本优劣判定上，优先采信大样本直接对战与固定对手池复验，而非单轮 gauntlet 名次。
15. 版本产出谨慎原则：
   - 默认先在现有版本上做“小步可回滚”改造，不得为“尝试一下”直接新建版本目录；
   - 仅当改动思路与现有版本结构明显不兼容，或已通过 gate 评测后需要固化快照时，才允许新建版本。
16. 旧 AI 借鉴强制项：
   - 每轮必须显式参考 `past_AIs/Generals-AI` 与 `past_AIs/ANTWar-AI` 至少各 1 个可落地机制；
   - 文档需写明“借鉴点 -> 新规则映射 -> 实现位置 -> 结果”，禁止只写泛化口号。
17. 简化优先原则：
   - 对等收益下，优先选择更短路径、更少状态变量、更少分支依赖的实现；
   - 若新增复杂逻辑导致收益不稳定，应优先回退复杂度，而不是继续叠补丁。

## 轮次交付定义

每次自动迭代至少完成以下之一：

- 新增一个可评测的 AI 版本分支（例如 `ai_cpp/v4`），并注册到 autolab；或
- 对现有版本做一次清晰的算法改造，并通过评测验证收益/退化；或
- 对评测框架做提升（更公平、更快、更稳）并验证有效。

## 失败处理

若本轮未能提升 Elo，也必须输出失败分析与下一轮假设，不允许“无结论结束”。
