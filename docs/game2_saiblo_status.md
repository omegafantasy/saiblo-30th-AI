## 1. 当前必须先统一的口径

`Game2` 不是传统双边对战。

对 `batch` 的正确理解是：

- 每个 `match` 都是“第一个 code 自己的一次独立跑分”
- 第二个 code 的 `score=0` 只是占位
- 因此分析 `Game2` 时，核心只看我方自己的 raw score
- 选择哪个“对手 code”，本质上只是为了更方便地拿到多份自跑样本

这意味着：

- 不应该再用 `win/loss` 口径理解 `Game2`
- `2357` 最初只是 `admin` 的参考上界样本，但 `2026-04-19` 的 `v43` 已在普通账号上复现同分

## 8. 下一步建议

后续如果继续迭代，优先级应改成：

1. 把 batch 当成“重复自跑采样器”，不再用对手强弱来解释分数。
2. 不要再为 `NPC_E/E`、`A/C/E` 顺序块、或 `python/auto` 运行类型继续花时间。
3. `v42` 已经排掉了“`answer` 之后机械补发两句隐藏尾部请求”这条线；后续不要再回到 post-answer 机械重放。
4. 重点盯这组已验证有效的 pre-answer 双问补丁，而不是继续堆 `case0` 细节。
5. `v44/v45/v46` 已经把“单问是否足够”和“顺序是否关键”都做空了；不要再回到 `C-only / D-only / 顺序反转` 这三类验证。
6. 不要再把“如何迁回真实主线”当成待办；这一步已经通过 `g2auto v27` 完成。
7. 如果还要继续压缩，只适合做极窄的“同义问句/措辞变体”实验：
   - 针对关键触发问句准备少量高针对性替换句
   - 根据回复关键词决定是否切换变体，而不是盲目重复问，或在尾部机械追加两句
   - 目标不是重新验证“有没有潜力”，而是确认是否存在同分的等价问法或更稳的措辞
8. 不要继续把“打印 `answer` 原始返回”当成高优先级方向；`v41` 已显示成功路径下 post-answer 玩家 `stderr` 不会稳定保留。
9. 普通账号之间的随机跨账号复测，以及“普通账号 + auto”组合复测，都已经被做空了；除非有新假设，否则没必要再重复。
10. 如果没有新的强假设，现在就可以把 `theend / g2auto v27` 视为 `Game2` 当前稳定版本。
11. 任何后续实验都至少做多次自跑样本验证，避免再次被单场结果误导。
