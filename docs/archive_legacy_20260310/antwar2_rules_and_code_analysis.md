## 2. 规则页核心信息（从 mhtml 解码）

规则页内容是“背景叙事 + SDK 接入说明”的混合体，核心可落地信息为：

- 回合制双人对战。
- 地图为 `row x col`，存在平原/沼泽/山地地形。
- 角色包含主将、副将、农夫。
- AI 回合内可执行移动、升级、技能、科技、超级武器、征召副将。
- 超过一定回合进入平局裁决（文案中同时出现 500 与 512 的描述）。
- SDK 与 AI 接口以 `Ant-Game` 仓库实现为准。

## 3. 代码中的真实规则（以 `logic/` 为准）

### 3.1 地图与初始

- 地图固定 `15x15`（`logic/constant.py`）。
- 地形随机生成：沼泽 15%，山地 5%，其余平原。
- 初始金币：双方 `50`。
- 初始单位：
  - 双方主将各 1 个，主将格初始兵力 50。
  - 中立副将 4 个。
  - 中立农夫 8 个。

### 3.2 指令体系（`execute_single_command`）

`logic/ai2logic.py` 支持 1~7 类操作：

- `1` 军队移动：`[1, x, y, dir(1-4), num]`
- `2` 将军移动：`[2, general_id, tx, ty]`
- `3` 将军升级：`[3, general_id, kind(1产量/2防御/3机动)]`
- `4` 战法：`[4, general_id, skill_id, ...]`
- `5` 科技：`[5, tech_id(1-4)]`
- `6` 超武：`[6, weapon_id, ...]`
- `7` 征召副将：`[7, x, y]`
- 回合结束：`[8]`

### 3.3 回合结算（`logic/gamestate.py::update_round`）

每个完整回合（双方都结束后）触发：

- 主将/己方副将产兵。
- 己方农夫产金币。
- 每 10 回合占领格 +1 兵。
- 沼泽减兵（未免疫科技时）。
- 激活中的超武持续效果结算与衰减。
- 将军技能 CD 与持续回合衰减。
- `rest_move_step` 重置。
- 双方金币 +1。
- 回放写入。

### 3.4 胜负判定（`logic/game_rules.py`）

- 任意主将被消灭：另一方立即胜。
- `round <= 500`：继续。
- `round > 500`：按 `总兵力 > 占格数 > 金币` 决胜。
- 若全相同，先手（玩家 0）胜。

## 4. 协议与评测链路

### 4.1 AI 进程通信（`main.py`）

- `judger -> logic`: `[4B length][json]`
- `logic -> judger`: `[4B length][4B target][payload]`
- `logic -> AI`（被转发后）为行文本：
  - 初始化行：`"<seat> <seed>"`
  - 状态行：单行 JSON（包含 `Cells/Generals/Coins/Tech_level/Turn` 等）
- `AI -> judger`: `[4B length][payload]`，payload 为文本命令串（多行，末行应为 `8`）。

### 4.2 本地评测链路

- 纯 Python AI：`play.py` / `logic/runner.py` 直接调用策略函数。
- C++ AI：通过 Python 桥接层把 `GameState` 转为协议 JSON，写入 C++ 进程，再读取其长度前缀返回命令。

## 5. 规则文案与代码的关键差异

### 差异 A：叙事仍包含 AntWar 塔防元素

规则页中有“防御塔、工蚁路径、EMP 干扰塔”等描述；但当前代码实现核心是“将军/军队占领格”模型（更接近 Generals）。

### 差异 B：结束回合阈值描述不一致

- 文案有“到 512 结束”的表述。
- 代码是 `round > 500` 进入平局裁决。

### 差异 C：在线主流程可能出现每回合额外 +1 金币

- `update_round()` 已做一次双方 `+1 coin`。
- `main.py` 在调用 `update_round()` 后又再次 `+1`。
- 这会使逻辑主进程路径与本地 `run_match()` 路径存在经济节奏差异。

### 差异 D：地形与移动是四方向网格，不是六边形

当前实现只有上下左右，且地图为 15x15 方格。

## 6. 对 AI 编写的直接结论

- 不要按“防御塔/工蚁自动寻路”假设来写策略。
- 当前可用且稳定的策略主轴应是：
  - 主将周边扩张
  - 抢中立副将/农夫
  - 金币管理（升级/科技/征召）
  - 500 回合前的兵力与占格累积
- 若要线上成绩与本地一致，需要优先核对“回合金币双加”是否为线上真实执行版本。

## 7. 后续建议（第二阶段）

- 增加“线上对局回放 vs 本地复盘”的自动比对脚本，重点校验：
  - 每回合金币增量
  - 超武 CD 与持续回合
  - 平局裁决触发回合
- 将策略搜索从单步贪心升级到 2~4 ply 快速前瞻（以回合预算为约束）。
