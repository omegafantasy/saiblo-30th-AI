## 单人房间评测方法

已确认 Game53 支持单人房间：

```bash
python3 Game2/tools/run_room_eval.py \
  --code-id <code_id> \
  --label <neutral_label> \
  --count 5 \
  --timeout 420 \
  --poll-interval 2
```

API 流程：

1. `POST /api/rooms/`，payload 为 `{"game_id":53,"player_number":1}`。
2. `POST /api/rooms/<room_id>/join/`，把我方 code 放在 `order=0`。
3. `POST /api/rooms/<room_id>/begin_match/`。
4. 拉取 match detail、download、analysis，并写入 `Game2/runtime/room_matches/`。

注意：

- `player_number=2` 对 Game53 会报 `游戏人数设置不正确`。
- `match_download.json` 当前只暴露 key `0`，后续案子的行为主要要从 `decoded_stdin_records` 与玩家 `stderr` 反推。
- 因为后续案子的 per-case answer_result 不可见，版本判断必须依赖多局 raw score 分布，而不是单局高分。

## 理论上限判断

按 SDK 文档，每个剧本分数由答案正确性最高 `600`、调查进度最高 `400`、成就 `40/个` 和效率系数组成，总分为各剧本求和。因此四个剧本的纯理论上限至少应远高于当前最高观测 `2707`。当前差距最大的位置不是 Rose，而是：

- 后两个新增案子仍几乎没有做完整调查；扑克案只证明到 stage2 就能多约 `50`，理论上 stage3+ 和袁案调查仍应有空间。
- Z/F 案当前已恢复旧隐藏双问并实测提升到 `2657`，但可见 stage 仍停在 `7`，stage8 通过平台 `KeyError('8')` 体现，无法继续正常深挖。
- 但平台评分含效率系数，盲目加聊天会降低效率且可能扰乱答案匹配；已完成测试显示扑克、袁案追加信息经常从 `2657` 掉到 `2617/2457/2417/2217`。
