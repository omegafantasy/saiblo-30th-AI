# Game1 Random Search Strategy

这份文档记录当前 Game1 C++ AI 的策略基线，直接对应当前仓库中的新规则与已确认实现方向。

说明：

- 这份文档描述的是“当前实现中的 baseline”
- 它不代表当前最优方向
- 基于手打 replay `Game1/good.json` 的下一阶段主方向，见 `docs/game1_lure_strategy.md`

目标不是复刻旧版 `heavy/bewitch` 脚本策略，而是把这些经验压进：

- 候选动作生成
- 轻量随机搜索
- 终点评估
- `hold` 偏置与经济/威胁权衡

所有历史自动化结果、旧规则快照、旧版细节策略都不再作为依据。

## 1. Scope

当前版本只做 Game1 防守决策，且严格以当前代码机制为准。

经济口径补充：

- 官方基础收入是每 `2` 回合 `+3`
- 不再沿用“每回合 `+3`”的旧假设

本阶段搜索中：

- 不考虑对手本回合和未来的主动操作
- 不考虑我方主动进攻操作
- 不考虑基地升级
- 不考虑 `EMP` / `Deflector` / `Emergency Evasion` 的主动使用
- 主防守搜索中忽略我方蚂蚁
- 仅额外考虑我方本回合主动释放 `Lightning Storm`
- 仅考虑当前已经激活的敌方超级武器效果

实现口径补充：

- 当前代码真值里，超武是在玩家操作阶段部署后立即生效
- 但本策略仍然不主动模拟“对手未来新放的超武”，只处理当前回合开始时已经可见的激活效果，以及我方本回合可能主动释放的闪电

当前允许的塔集合：

- `Basic`
- `Heavy`
- `Mortar`
- `Quick`
- `Bewitch`
- `QuickPlus`

当前允许的动作集合：

- `Hold`
- `BuildTower`
- `UpgradeTower`
- `DowngradeTower`
- `UseLightningStorm`

## 2. 总体决策框架

每回合决策分三层：

1. 先计算一次共享的进攻收益估计
2. 再对一组有限的防守候选计划做轻量随机搜索
3. 如有必要，单独做一次闪电子搜索，补充本回合闪电候选

最终从以下计划中选最优：

- 仅当前回合动作
- 当前回合动作 + 下一回合预定动作

这里的“两回合计划”只是当前回合决策时的前瞻，不代表第二回合一定执行；它本质上是在搜索里评估“当前操作是否能为下一回合创造高价值延续”。

## 3. 两回合动作约束

当前 baseline 的动作模型仍做了强简化：把一回合近似成“至多一个核心操作”，因此搜索只考虑：

- `0` 个操作
- `1` 个操作
- 连续 `2` 回合各 `1` 个操作

这不是当前官方规则真值；官方 engine 实际支持同回合有序操作列表。

两回合计划的约束如下：

- 若第一回合是建塔，则第二回合只能升级刚建成的那一座塔
- 当前只在核心九格 `C1/C2/C3/L1/L2/L3/R1/R2/R3` 上生成这类 `Build -> Upgrade`
- 若第一回合是拆塔或降级，则第二回合可以是别的合法操作，但不能升级/重建刚处理过的那个格子
- 当前若 `Downgrade` 后第二回合选择建塔，也只允许核心九格
- 若第一回合是升级，则不考虑第二回合操作
- 若第一回合是闪电，则第二回合不允许操作
- 当前阶段不考虑任何基地升级

实现上，第二回合操作以“条件计划”处理：

- 先记录计划意图
- rollout 推进到下一回合时，再依据当时的金币、塔状态、EMP 覆盖等条件检查是否仍合法
- 若不合法，则该 rollout 中第二回合自动退化为 `Hold`
- 第二回合操作不是“根据 rollout 临时重规划”，而是“预先承诺一条固定 second op，再逐 rollout 检查能否执行”

## 4. 主防守搜索状态

主防守搜索只保留和防守直接相关的部分：

- 我方基地血量
- 我方金币
- 我方塔
- 敌方蚂蚁
- 我方生效中的 `Lightning Storm`
- 敌方当前已激活的超级武器
- 敌方基地的自然产兵节奏

明确忽略：

- 我方当前蚂蚁
- 我方后续自然产兵的直接影响
- 对手未来主动操作
- 大策略硬编码

设计意图：

- 我方蚂蚁的进攻价值不并入主防守模拟，改为单独做共享的进攻 EV
- 防守模拟专注于“现有敌方压力 + 当前塔与经济”的应对

## 5. 共享进攻 EV

我方进攻价值单独计算一次，然后被所有防守候选共享。

计算方式：

- 从当前真实局面出发
- 忽略双方未来操作
- 仅统计我方未来 `6` 回合的金币收益期望
- 当前实现使用 `128` 次随机模拟
- 每次使用不同随机种子/随机轨迹

收益定义只关心我方钱，不关心敌方基地血量。

这部分收益按“逐回合期望增量”并入主防守搜索中的我方金币，而不是最后一次性补到终点评分里。

这意味着主防守搜索里的钱近似等于：

- 当前真实金币
- 当前计划动作直接带来的花费/返还
- 官方基础收入：每 `2` 回合 `+3`
- 独立进攻 EV 带来的逐回合期望收益

## 6. 候选动作生成

候选动作不追求全图穷举，而是高质量、小规模、强约束。

### 6.1 建塔

建塔候选格优先来自：

- `strategic_slot_order`
- 基地前线关键格
- 当前高威胁蚂蚁附近的拦截格
- 现有高价值塔前后的保护格

当前仅生成这些建塔目标：

- 建 `Basic`

后续若进入两回合计划，允许第二回合把这座 `Basic` 升到：

- `Heavy`
- `Mortar`
- `Quick`

但这类 `Build -> Upgrade` 当前只允许出现在核心九格。

### 6.2 升级

当前回合允许的升级候选只保留低复杂度、高相关升级：

- `Basic -> Heavy`
- `Basic -> Mortar`
- `Basic -> Quick`
- `Heavy -> Bewitch`
- `Quick -> QuickPlus`

不考虑其他升级分支。
同时，`Upgrade` 当前只作为单回合动作，不再接第二回合 followup。

### 6.3 降级/拆塔

优先生成以下塔的降级/拆除候选：

- 可能很快被战斗蚁拆掉的高价值塔
- 基地门口阻塞后续更优建塔的低价值塔
- 已明显处于亏损位置、保不住的塔

### 6.4 闪电

闪电单独作为候选，并额外走一套独立位置搜索。

位置生成规则：

- 优先覆盖高密度敌蚁区域
- 优先覆盖靠近基地或高价值塔的威胁团
- 排除距离地图边界小于 `3` 的中心

### 6.5 保留强 `Hold`

`Hold` 永远作为强基线候选。

因为当前版本下大多数频繁操作都会导致严重经济损失，所以搜索仍然默认偏向“不动”，只有估值确认收益明显更大时才做动作。

## 7. 主搜索 Rollout

每个防守候选计划不是一开始就直接打满 `100` 次，而是按 progressive allocation 分配预算：

- 先给所有候选各做 `20` 次
- 再把更优候选按批次继续补 `20`
- 单个候选最高补到 `100`

每次 rollout：

- 前进 `6` 回合
- 使用独立随机轨迹
- 模拟正常移动与每 `10` 回合一次的随机移动阶段
- 只在第一回合显式考虑“重要蚂蚁第一步动作覆盖”

rollout 只看终点状态做评估，不做中间截断评分。

这里的 rollout / legality 判断默认与最新规则对齐：

- 同回合后手可能被前手刚放下的 `EMP` 直接卡掉塔操作
- 我方本回合释放 `Lightning Storm` 时，其即时效果应计入该 rollout

## 8. 第一回合蚂蚁动作覆盖

纯随机采样会漏掉稀有但高威胁的首步组合，因此对第一回合做受控覆盖。

### 8.1 只覆盖少量关键蚂蚁

最多只选 `3` 只最重要的敌蚂蚁进入显式覆盖。

威胁排序优先考虑：

- 战斗蚁高于普通蚁
- 离我方基地越近越高
- 离我方高价值塔越近越高
- 血量越高越高
- 已 `Randomized` / `Bewitched` 的威胁适当下调

### 8.2 只覆盖第一步

显式覆盖只作用于“第一回合第一步”的动作组合。

后续所有步、以及其余蚂蚁的行动全部按随机过程自然推进。

### 8.3 组合覆盖目标

覆盖策略同时满足两点：

- 高频组合必须按概率大量出现
- 稀有但高威胁组合必须至少被硬覆盖到

实现原则：

- 为每只关键蚂蚁构造少量首步候选与近似概率
- 只枚举这些蚂蚁的首步组合
- 组合数超预算时，按“组合威胁度 + 稀有补偿”裁剪
- 全部 rollout 硬上限 `100`

### 8.4 权重

每个被覆盖的首步组合都带一个近似组合概率。

最终候选计划分数使用“组合概率加权平均”的方式聚合。

这保证：

- 高频场景不会因为硬覆盖而被低估
- 极少见但危险的场景也不会完全漏掉

## 9. 终点评估

rollout 终点评估只使用以下项：

- 我方基地血量
- 我方塔的剩余价值
- 我方塔的类型/位置奖励
- 敌方蚂蚁威胁值
- 我方当前钱
- `Hold` 偏置

当前实现直接使用一套简化分数：

`Score = 200 * base_hp + 10 * coins + 10 * tower_salvage + tower_bonus - ant_threat + (hold ? 50 : 0)`

### 9.1 基地血量

这是最强权重项，单位权重固定为：

- `1` 点基地血 = `200`

搜索窗口只有 `6` 回合，因此“逼近基地但尚未进家”的蚂蚁风险，不直接体现在 `base_hp` 里，而是转移到下方的 `ant_threat`。

### 9.2 塔剩余价值

塔价值主项按“当前若全部回收，理论还能拿回多少钱”衡量。

这部分应考虑：

- 当前塔级别对应的建造/升级成本
- 当前血量比例
- 当前规则的返还公式

设计意图：

- 同样是塔，保住一个能退回大量金币的高血量塔，比保住一个残血低价值塔更重要
- 这也自然鼓励在危险时机降级/拆除返现

主项的单位权重固定为：

- `1` 金币 = `10`

此外，当前还保留两类塔奖励：

- `Heavy = 30 * hp_ratio`
- `Bewitch = 100 * hp_ratio`

位置奖励按旧版九格标记：

- `C1 = 30`
- `C2 = 25`
- `C3 = 20`
- `L1 / R1 = 16`
- `L2 / R2 = 10`
- `L3 / R3 = 8`

位置奖励同样乘以塔当前 `hp_ratio`。

### 9.3 敌蚂蚁威胁值

所有蚂蚁都先计算一项基础基地威胁：

- `200 * distance_factor * hp_ratio * behavior_factor`

其中 `distance_factor` 当前使用：

- `d=0: 1.00`
- `d=1: 0.80`
- `d=2: 0.60`
- `d=3: 0.42`
- `d=4: 0.28`
- `d=5: 0.18`
- `d=6: 0.10`
- `d>=7: 0.05`

行为因子当前使用：

- `default / conservative / control_free = 1.0`
- `random = 0.6`
- `bewitched = 0.25`

战斗蚁额外再计算一项“对塔的钱威胁”：

- `tower_salvage * 0.3 * 10 * decay * hp_ratio * behavior_factor`

当前距离衰减为：

- 距塔 `1`：`1.00`
- 距塔 `2`：`0.60`
- 距塔 `3`：`0.35`
- 距塔 `4`：`0.20`
- 距塔 `5`：`0.10`

这项额外塔威胁可以让单只战斗蚁的总威胁超过 `200`。

### 9.4 金币

保留钱本身就是未来防守能力，因此终点评估要给钱正权重。

- `1` 金币 = `10`

它既奖励有效击杀带来的收益，也会自然惩罚无必要的建塔、升级、闪电等花钱操作。

## 10. 动作惩罚

当前实现已经去掉大多数显式动作惩罚。

现在直接生效的“偏向不动”项只有：

- `Hold` 额外 `+50`

少操作倾向主要来自以下几项的共同作用：

- 花钱操作会直接减少 `coins`
- 建塔/升级会改变未来塔的可回收价值结构
- 战斗蚁对高价值塔的威胁会被显式估值
- `Hold` 固定有正偏置

这意味着当前版本不是“硬编码惩罚一切操作”，而是让经济与威胁结构自己决定动作是否值得。

## 11. 闪电子搜索

闪电不应默认进入主策略节奏，而更接近一个需要独立评估的应急动作。

为避免把大量主搜索预算浪费在闪电位置上，闪电单独做一次轻量子搜索：

- 仅考虑本回合使用闪电
- 当前最多评估 `6` 个候选中心
- 每个中心当前做 `18` 次重复取平均
- 每次前进 `15` 回合
- 枚举或采样一组候选中心
- 用更轻量的终点评估比较覆盖效果

闪电评分偏好：

- 覆盖高密度敌蚁
- 覆盖靠近基地的危险团
- 覆盖正在威胁高价值塔的战斗蚁

当前实现里，闪电不再额外叠加专门的高惩罚项；它是否值得使用，主要由金币损失与终点 threat 下降之间的权衡决定。

## 12. 与旧策略经验的关系

此前关于：

- 前期空城
- 基地门口 `Heavy`
- 再转 `Bewitch`
- `Quick` 保护
- 战斗蚁逼近时主动返现

这些判断仍然有参考价值，但不再以脚本形式硬编码。

新的写法原则是：

- 不直接写“到了某回合一定做某事”
- 不把某一套塔序列当作硬规则
- 尽量把这些经验翻译成候选动作与估值偏置

这样：

- `Mortar` 这类在某些局面意外有效的塔，也能被搜索选中
- 后续调参时只需要改估值与动作生成，不必反复重写大段策略脚本

## 13. 实现阶段

当前实现分三步：

1. 先完成文档与代码框架替换
2. 落地可运行的轻量随机搜索基线
3. 之后再围绕速度、首步覆盖质量、估值参数、候选位置做迭代

首版实现优先保证：

- 搜索结构正确
- 约束正确
- 动作空间正确
- 终点评估方向正确
- 能稳定构建和运行

后续重点优化：

- 更高质量的首步覆盖
- 更快的轻量模拟
- 更强的塔位筛选
- 更稳的闪电子搜索
- 行为/威胁估值参数调优

## 14. 复算与校验口径

当前基线为了速度，主搜索并不直接调用原生 `NativeSimulator` 做防守 rollout，而是使用外置轻量 `DefenseSimulator` 近似模拟。

这意味着：

- 搜索分数可以作为决策参考
- 但不能直接当成规则真值或行为真值
- 一旦发现“明显亏钱但搜索强推”的动作，必须回到原生 replay 复算确认

当前轻量模拟已经落地的主要提速手段：

- 高频塔查询改成 O(1) 占位缓存
- 静态风险场 / move phase cache / tower-path cache 按需构建
- 无 combat 蚂蚁时不构建 combat tower path
- 无 enhanced move 需求时不预热整套 move cache

当前建议的复算方法：

- 先用 replay 从 `round 0` 精确回放到目标回合，保留隐藏态
- 再从该回合状态分叉不同未来随机种子
- 直接比较 `hold` 与指定 action 的 `1000+` 次原生 Monte Carlo 均值
- 不再使用 bandit 分配或候选剪枝，直接做纯复算

当前仓库已提供对应工具：

- `Game1/antgame_cpp_sdk/examples/sdk_rollout_probe.cpp`
- 可同时输出：
  - 轻量 `DefenseSimulator` 的 Monte Carlo 估值
  - 原生 `NativeSimulator` 的 Monte Carlo 估值

当前已确认的关键案例：

- `seed_0024`, `player 0`, `round 199`
- 候选动作：`hold` vs `build 11:4:9`

在该案例中：

- 轻量防守模拟会把 `build 11:4:9` 评得远高于 `hold`
- 在旧的 `4` 回合窗口配置下，它错误地预测该塔能多保约 `2.8` 点基地血
- 原生 replay 复算下，这个动作对基地血的真实增益只有约 `0.1`
- 因此这个 case 的主要问题不是 rollout 分配，而是轻量模拟本身存在明显高估

当前调试优先级应为：

1. 先修轻量模拟与原生行为的偏差
2. 再调 rollout 分配、bandit 策略和参数

## 15. 当前默认假设

为了尽快形成可运行基线，当前实现默认采用以下工程假设：

- 主防守搜索会纳入敌方基地的自然产兵
- 不主动模拟对手未来新释放的超级武器
- 不主动模拟我方未来新产蚁的防守作用
- 旧版 `heavy_baseline` 的脚本逻辑全部视为已废弃

若后续策略讨论确认其中某项需要调整，应直接改这份文档和对应实现，不再维护并行版本。
