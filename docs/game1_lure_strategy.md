# Game1 Lure Strategy

这份文档记录基于手打 replay `Game1/good.json` 总结出的下一阶段主策略方向。

player 0 为手打方。核心思想不是“尽量全防普通蚂蚁”，而是：

- 先想办法建立 `C1` 永久锚点
- 把主要精力放在战斗蚁控制上
- 用外圈廉价 `Basic` 塔反复引诱战斗蚁偏离基地
- 在战斗蚁贴塔前及时拆塔回收
- 必要时结合 `Lightning Storm` 一次性清理积累的战斗蚁

这是一套强启发式搜索策略，不是固定脚本。

## 1. Replay Backed Facts

`good.json` 中可以直接看到以下事实：

- 该 replay 的塔快照在少数回合存在抖动，因此本页结论主要基于：
  - `op0` 操作序列
  - 基地血量与金币曲线
  - 战斗蚁轨迹
- 对“某一帧塔是否显示出来”的判断，不作为主要依据

- `R11`：`C1` 建出 `Basic`
- `R18`：`C1 -> Heavy`
- `R41-R73`：第一段外圈引诱链，依次用到 `F3 / M1 / F2 / ML1 / F2`
- `R65`：出现关键三连操作：`拆 M1 -> Lightning@(8,6) -> 建 F2`
- `R69-R70`：该波闪电后，战斗蚁 `23 / 7 / 11` 依次死亡
- `R101-R105`：战斗蚁逼近 `C1`，先卖掉旧 `C1`，再重建并转入 `Quick` 路径
- `R119`：`卖 M4 -> C1 Quick -> Sniper`
- `R133-R163`：`M1 / M4 / M3` 继续承担引诱位，配合第二次闪电
- `R203-R214`：后期出现 `M2 / M1 / M4` 与 `L1 / C2` 的近基地网，第三次闪电后转入双 `Quick`
- `R242`：`L1 Quick -> Sniper`

本局 player 0 的建塔位置高度集中：

- `M1` 共建 `9` 次
- `M4` 共建 `7` 次
- `F2` 共建 `3` 次
- `F3 / ML1 / M3 / M2 / C3 / L1 / C2` 各 `1` 次

这说明外部引诱位不是全图泛化，而是少量高价值槽位反复轮换。

本局 player 0 共进行了：

- `26` 次降级/拆塔
- `3` 次闪电
- 多次同回合链式操作

这也说明“单动作 / 固定两回合 followup”不足以表达该策略。

## 2. Why The Strategy Works

这套策略的经济基础很清楚：

- 当 `C1` 作为唯一永久塔存在时，额外建一个诱饵 `Basic` 的价格通常是 `30`
- 诱饵在满血时拆掉，返还通常是 `27`
- 因此一次“建诱饵再安全卖掉”的净成本通常只有 `3`

而战斗蚁的收益很大：

- 会明显偏向拆塔
- 不会老死
- 若不主动引走，就会持续在基地附近形成高压
- 一旦积累，后续很适合被闪电成批处理

所以这套策略的核心不是“用塔把战斗蚁打死”，而是：

- 用很小的经济损耗换取战斗蚁路线控制
- 把战斗蚁长期控制在离基地更远的位置
- 让它们尽量不要去碰 `C1`
- 等积累到合适规模后，用闪电收割

这也是为什么前期可以接受卖血：

- 普通蚂蚁的威胁多数只是固定 `-1` 基地血
- 开局经济太弱时，试图全防普通蚂蚁通常会导致更差的总经济
- 只要战斗蚁被稳定控制，中后期经济会反而更健康

`good.json` 里也能看到这个结果：

- `R121` 起 player 0 基地血稳定在 `36`
- 此后一直到 replay 结束都没有继续掉血
- 金币却还能继续滚到 `301`

这说明该策略虽然前期会掉血，但一旦进入稳定态，整体强度明显高于当前 baseline。

## 3. Strategic Thesis

这套“引诱战略”可以抽象成以下几条硬原则：

- 优先处理战斗蚁，不优先处理普通蚂蚁
- `C1` 是主锚点，但不是不可出售的圣塔
- 诱饵塔是消耗品，不是永久资产
- 诱饵的价值主要来自“改变战斗蚁路线”，不是输出
- 闪电不是独立手段，而是“引诱后的收束工具”

更具体地说：

- 早期目标是先把 `C1` 固化成 `Heavy`
- 当经济稳定后，再转入 `Quick -> Sniper`
- 若战斗蚁已经威胁到 `C1`，应立刻接受“放弃 `C1`”而不是恋战
- 外部诱饵位应尽量放在中线附近和外圈高地，而不是基地口
- 诱饵一旦进入“下回合会被战斗蚁白拆”的状态，就应该优先卖掉
- 闪电的点位不只看当前敌蚁位置，还要结合后续诱饵位，让战斗蚁不容易逃出范围

## 4. Position Pattern From `good.json`

这局回放实际使用的主要位置可以分成三层：

- 锚点层：`C1`
- 外圈引诱层：`F3 / F2 / ML1 / M1 / M4 / M3 / M2`
- 近基地补网层：`C3 / L1 / C2`

这给出一个很直接的实现建议：

- 不要再把候选建塔位置主要限制在核心九格
- 应该显式维护一组“引诱槽位表”
- 搜索时优先在这些槽位上做建塔/卖塔/闪电联动

后续应至少准备两组镜像槽位：

- 上侧引诱链
- 下侧引诱链

并允许在链上做“卖旧诱饵 -> 建新诱饵”的跳转。

## 5. What The Current Baseline Misses

当前 baseline 与这套策略有几处本质错位：

- 当前主要按“单动作 + 固定 second op”建模
- 但官方规则实际支持同回合有序操作列表
- 当前 rollout 未来动作基本不是自适应策略，而是固定 followup
- 当前候选位置更偏“守家塔”，而不是“外圈引诱位”
- 当前估值更像“局部短 horizon 防守分”，而不是“战斗蚁控制分”

这会直接导致几个问题：

- 无法表达 `拆塔 -> 建塔`
- 无法表达 `拆塔 -> 闪电 -> 建塔`
- 无法在模拟里做到“战斗蚁近身就自动卖诱饵”
- 无法正确利用“保持塔数低，从而让下一座诱饵仍然只要 `30`”

因此，后续不应只在当前 action 模型上微调参数。

## 6. Recommended Algorithm

推荐采用“根节点搜索 + 未来回合反应式控制器”的混合框架。

核心思想：

- 当前回合仍然做随机搜索
- 但搜索对象不再是单操作，而是“本回合有序 op-list”
- rollout 进入未来回合后，不再沿用固定 second op
- 而是每回合调用一次轻量的 `reactive_lure_policy()`，在模拟态上自适应地产生新的 op-list

这样可以同时兼顾：

- 搜索
- 强启发式
- 多操作链
- 近身自动卖塔

## 7. Turn Action Should Become `OpList`

新的动作单位应是：

- `[]`
- `[Sell lure]`
- `[Build lure@slot]`
- `[Sell lure, Build lure@slot]`
- `[Sell lure, Lightning@center]`
- `[Sell lure, Lightning@center, Build lure@slot]`
- `[Sell C1]`
- `[Build C1]`
- `[Upgrade C1->Heavy]`
- `[Upgrade C1 Quick->Sniper]`
- `[Sell two lures, Build two close towers]`

这里的关键不是把所有组合全暴力穷举，而是：

- 先按模板生成少量“有语义的 op-list”
- 再让 rollout 用反应式控制器接管未来回合

## 8. Reactive Lure Policy

rollout 中未来每一回合，建议按如下顺序执行一个廉价控制器：

1. 先检查 `C1` 是否已经不安全
2. 再检查所有诱饵塔是否进入“应立即卖掉”的区间
3. 再决定是否放闪电
4. 最后决定是否在新的引诱槽位建一个新的 `Basic`
5. 若经济允许且局面稳定，再考虑 `C1` 的重建或升级

其中“应立即卖掉”的判定不应只看当前是否相邻，还应看：

- 下回合战斗蚁是否大概率贴塔
- 当前塔是否只是诱饵而非永久塔
- 该塔若不卖，是否会把本轮经济直接亏成更高建塔档位

也就是说，卖塔不是被动补救，而是主动策略动作。

## 9. Root Candidate Generation

当前回合的候选 op-list 建议按以下方式生成：

- `Hold`
- `Anchor ops`
- `Lure relocate ops`
- `Lightning ops`
- `Lightning + lure ops`
- `Late-game close-net ops`

更具体地：

- `Anchor ops` 只围绕 `C1`
- `Lure relocate ops` 只围绕少量外圈/中圈槽位
- `Lightning ops` 只保留少量高价值中心
- `Late-game close-net ops` 只在高经济、且普通蚂蚁压力较大时开放

推荐的根节点生成顺序：

1. 先生成所有“卖塔集合”
2. 再在卖塔后的钱包与塔数基础上生成闪电候选
3. 再生成建塔候选
4. 最后生成少量升级候选

这样可以天然利用“先卖再建”的成本优势。

## 10. Evaluation Direction

新的估值不应再把所有塔都视为越多越好，而应区分：

- 永久锚点塔
- 临时诱饵塔

建议的核心估值项：

- 基地血量
- 当前金币
- 所有塔的可回收价值
- `C1` 锚点奖励
- 战斗蚁控制分
- 闪电覆盖分
- 普通蚂蚁近基地威胁

其中“战斗蚁控制分”应成为核心新项，至少要体现：

- 距基地越近越危险
- 距 `C1` 或其他高价值永久塔越近越危险
- 若正被外圈诱饵吸住，则危险下降
- 若正处于闪电范围且不易逃出，则危险进一步下降

对普通蚂蚁则应明显降权：

- 早期允许漏怪
- 中后期只有在 `C1 Sniper` 尚未成型时，才适度提高权重

## 11. Immediate Refactor Targets

若要真正实现该策略，建议按以下顺序改：

1. 把搜索动作从单操作改为 `OpList`
2. 加一个顺序合法性与顺序花费模拟器
3. 建立正式的“引诱槽位表”与镜像关系
4. 在 rollout 中加入 `reactive_lure_policy()`
5. 把估值从“守家”改成“战斗蚁控制 + 经济”
6. 再把闪电与引诱联合搜索并进来

这里最关键的不是数值微调，而是第 `1-4` 步。

## 12. Bottom Line

`good.json` 给出的结论不是“某几个点位很强”，而是：

- Game1 当前版本真正强的，不是纯防守塔阵
- 而是“永久锚点 + 廉价诱饵 + 主动回收 + 闪电收束”的控制型玩法

因此下一阶段的 AI，不应再围绕“如何少量建塔把所有怪守住”设计。

更合理的目标是：

- 用搜索驱动一套强启发式的战斗蚁控制器
- 允许每回合多操作链
- 允许在 rollout 中自适应卖塔、换点、闪电与重建

这才是当前规则下更接近可扩展强策略的方向。
